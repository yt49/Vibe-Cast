## 2. ゲストの声が録音されない問題を修正

### 問題の本質

元々のアプリにはゲストの録音機能が**まったく存在しなかった**。

ゲスト側の画面には「録音はホストが操作します」とだけ表示され、
ゲスト自身はマイクの録音を一切していなかった。

つまり：
- ホスト録音 → ホストのマイク音声のみ → GCSにアップロード ✅
- ゲスト録音 → **何もない** ❌

### 解決策：WebSocketでホストの操作をゲストに伝える

**新しく追加したWebSocketメッセージの種類：**

| メッセージタイプ | 送信者 | 意味 |
|---------------|-------|------|
| `role` | 双方 | 接続時に自分のロールを通知 |
| `session_id` | ホスト | 録音開始。このIDで録音してね |
| `session_stop` | ホスト | 録音停止してね |

**ゲスト自動録音の流れ：**

```
ホストが「録音開始」ボタンを押す
  ↓
ホストが session_id を WebSocket で送信
  ↓
ゲストの useWebRTC が受信 → setEpisodeId(id) でstate更新
  ↓
useEffect([episodeId]) が発火
  ↓
ゲストのマイクで start(localStream) を呼び出し → 録音開始！

ホストが「録音停止」ボタンを押す
  ↓
ホストが session_stop を WebSocket で送信
  ↓
ゲストの useWebRTC が受信 → setGuestShouldStop(true) でstate更新
  ↓
useEffect([guestShouldStop]) が発火
  ↓
stop() → upload(blob, 'guest', episodeId) → GCSにアップロード！
```

**なぜ useEffect を使うのか（コールバックで直接呼べばいいのでは？）：**

WebSocketのコールバック（`onmessage`）は**古い状態を参照してしまう**問題がある
（JavaScriptの「stale closure（古いクロージャ）」と呼ばれる現象）。

Reactの `useEffect` は state の変化を検知してから実行されるため、
常に最新の `localStream` や `remoteStream` を使って録音を開始できる。

### 変更したファイル

**`frontend/src/hooks/useWebRTC.ts`**
- `onRecordingStop` コールバックを追加
- `onRoleConflict` コールバックを追加
- `session_stop` メッセージの受信処理を追加
- `sendRecordingStop()` 関数を追加
- ref パターンでコールバックの stale closure を防止

**`frontend/src/pages/RecordingPage.tsx`**
- `useEffect([episodeId])` でゲストの自動録音開始
- `useEffect([guestShouldStop])` でゲストの自動録音停止
- `guestRecordingStartedRef` で二重起動を防止

## 3. 同じロールが2人つないでしまう問題を修正

### 問題

接続時にロールチェックをしていなかったため、
ホスト2人・ゲスト2人でも普通に接続できてしまっていた。

### 解決策

接続時に WebSocket で `{"type": "role", "role": "host"}` を送り合い、
相手と同じロールだった場合にエラーを表示する。

```
接続ボタンを押す
  ↓
WebSocket で {"type": "role", "role": "host"} を送信
  ↓
相手からも {"type": "role", "role": "host"} が届く
  ↓
role === 自分のrole → 「ロールが重複しています」エラーを表示
```

エラーが出たらロールを変えて再接続すればOK。

## 4. 録音ファイルのミックス

### 問題

修正前：ホストはホストの声だけ、ゲストはゲストの声だけ録音されていた。

Local Playback（ローカル再生）では**自分の声しか聞こえない**。

### 解決策：AudioContextでストリームをミックス

`AudioContext` はブラウザ内蔵の音声処理エンジン。

```typescript
function mixStreams(local: MediaStream, remote: MediaStream | null): MediaStream {
  const ctx = new AudioContext();
  const dest = ctx.createMediaStreamDestination();
  ctx.createMediaStreamSource(local).connect(dest);   // 自分の声
  ctx.createMediaStreamSource(remote).connect(dest);  // 相手の声
  return dest.stream;  // ミックスしたストリームを返す
}
```

このミックスしたストリームを `MediaRecorder` に渡すことで、
**両者の声が1つのファイルに録音**される。

## 5. 録音シークバーが動かない問題を修正

### 問題

`MediaRecorder` で生成した `.webm` ファイルには**録音時間の情報がない**。

ブラウザの `<audio>` 要素は時間情報がないと「全体の長さ不明」と判断してしまい、
シークバーを正しく操作できなかった。

### 解決策：`fix-webm-duration` ライブラリ

```bash
npm install fix-webm-duration
```

録音終了時に、測定していた録音時間をBlobのメタデータに書き込む：

```typescript
const duration = Date.now() - startTimeRef.current;  // 録音時間（ミリ秒）
fixWebmDuration(rawBlob, duration, (fixedBlob) => {
  // fixedBlob にはdurationメタデータが付いている
  resolve(fixedBlob);
});
```

これでシークバーが正しく機能するようになった。

## 今日のトラブルと教訓

### トラブル1：AudioContextでuseRecorderを改造したら無音になった

**やってしまったこと：**
`useRecorder.ts` の `start(stream)` を `async start()` に変えて、
内部で `getUserMedia()` を呼ぶように改造した。

**結果：**
呼び出し元（RecordingPage.tsx）は `start(localStream)` と呼んでいたが、
`localStream` 引数を無視して新たに `getUserMedia()` を呼ぶようになってしまい
AudioContextの設定が食い違って完全無音になった。

**修正：**
`useRecorder.ts` を git の `1a7ad65` コミット時点に戻し、
AudioContextのミックスは RecordingPage.tsx 側の `mixStreams()` 関数でやることにした。

**教訓：** フックの引数シグネチャを変えると呼び出し元との契約が壊れる。変更は最小限に。

### トラブル2：useEffect の依存配列に remoteStream を入れ忘れた

ゲストの自動録音 `useEffect` を最初に書いたとき、
依存配列が `[episodeId, role, localStream]` だけだった。

`remoteStream`（相手の音声）は `mixStreams()` に渡すのに依存配列にないため、
`remoteStream` が届く前に `useEffect` が走ってしまい、
ミックスされずローカル音声だけになる可能性があった。

`[episodeId, role, localStream, remoteStream]` に修正して解決。

```
  │

Cloud Run（FastAPI）
  ├─ /ws/{room_id}    → WebSocketシグナリング（role/session_id/session_stopを中継）
  ├─ /audio/upload    → GCS保存
  ├─ /ai/transcribe   → Whisper API
  └─ /ai/summarize    → Gemini API
```
