## 1. 音量メーター実装（#7）

### 何をやったか

マイクで話しているときに、画面上にリアルタイムで音量バーが表示されるようにした。
「話してるかどうか」を視覚的に確認できるようにするための機能。

### 仕組み（専門用語解説つき）

**Web Audio API** というブラウザ内蔵の音声処理ライブラリを使う。

```
マイク音声
  ↓
AudioContext（音声処理の司令塔）
  ↓
AnalyserNode（音を分析する機械）
  ↓
周波数データを取得（0〜255の数値の配列）
  ↓
平均値を計算して0〜1の数値に変換
  ↓
バーの幅に反映して画面に表示
```

**AnalyserNode** とは：
マイクの音声を「周波数ごとの強さ」に分解してくれる部品。
音楽イコライザーの表示（棒グラフが動くやつ）と同じ原理。

**色分けのルール：**
- 🟩 緑（音量 0〜40%）：静か
- 🟧 オレンジ（40〜70%）：普通
- 🔴 赤（70%〜）：大きい

### 作ったファイル

- `frontend/src/hooks/useVolumeLevel.ts`（新規）
  → 音量を計算するロジックをまとめたカスタムフック
- `frontend/src/pages/RecordingPage.tsx`（音量バーUI追加）

## 2. Whisper文字起こし実装（#11）

### 何をやったか

GCSに保存されたホスト・ゲストの音声ファイルを、OpenAI Whisper APIに送って文字起こしする機能を追加した。

### Whisperって何？

OpenAI（ChatGPTを作った会社）の音声認識AI。
音声ファイルを渡すと日本語テキストを返してくれる。
精度が非常に高い。コストは **$0.006/分**（1時間収録でも約¥55）。

### 仕組み

```
ユーザーが「文字起こし」ボタンを押す
  ↓
ブラウザ → Cloud Run（/ai/transcribe）にリクエスト
  ↓
Cloud Run → GCSからホスト・ゲストの音声ファイルをダウンロード
  ↓
Cloud Run → OpenAI Whisper APIに音声ファイルを送信（日本語指定）
  ↓
テキストが返ってくる
  ↓
ブラウザの画面に表示
```

### 話者分離をしない理由

「話者分離」とは、1つの音声ファイルから「どこが誰の発言か」を識別する技術。
複雑でコストもかかる。

今回は **最初から別々に録音している**（ホスト側・ゲスト側でそれぞれ録音）ので、
最初から別ファイルになっている。話者分離は不要。

### 作ったファイル

**`backend/app/services/transcriber.py`（新規）**

```python
def transcribe_episode(episode_id: str) -> dict:
    for role in ("host", "guest"):
        # GCSからファイルをダウンロード
        audio_bytes = _download_from_gcs(f"episodes/{episode_id}/raw/{role}_track.webm")

        # 一時ファイルに書き出してWhisperに渡す
        with tempfile.NamedTemporaryFile(suffix=".webm") as f:
            f.write(audio_bytes)
            response = openai_client.audio.transcriptions.create(
                model="whisper-1",
                file=f,
                language="ja",  # 日本語指定
            )
        results[role] = response.text
```

**`backend/app/routers/ai.py`（新規）**

FastAPIのルーター（URLのエンドポイントを定義する場所）。
- `POST /ai/transcribe` → 文字起こし
- `POST /ai/summarize` → Gemini要約（後述）

**`backend/requirements.txt`** に `openai` を追加。  
**`backend/app/main.py`** に aiルーターを登録。  
**`frontend/src/pages/RecordingPage.tsx`** に文字起こしボタンのUIを追加。

## 3. Gemini要約実装（#12）

### 何をやったか

Whisperで取れた文字起こしテキストを、Gemini APIに渡して構造化する。
「二人で作る日記」コンセプトの核心部分。

### Geminiって何？

GoogleのAI（ChatGPTのGoogle版）。
プロンプト（指示文）を渡すとテキストを生成してくれる。
今回はJSONフォーマットで返すよう指示している。

### 出力形式

```json
{
  "theme": "今回のテーマ",
  "host_claim": "鶴岡の主張（100字以内）",
  "guest_claim": "大悟の主張（100字以内）",
  "agreed": "合意したこと",
  "disagreed": "まだ対立していること",
  "host_question": "鶴岡が次回までに考えたい問い",
  "guest_question": "大悟が次回までに考えたい問い"
}
```

### 実際に動いた結果（テスト）

1人でテスト収録したときの出力：

```
テーマ：VIVEcastアプリのテストと現状の課題
鶴岡の主張：VIVEcastアプリのテスト中。Whisperの文字起こしは高精度だが、
            Geminiの要約機能が機能せず、テストの成否が不透明。
鶴岡の問い：VIVEcastのGeminiによる要約機能は今回のテストで成功するのか？
```

ちゃんとJSONで返ってきて、画面に表示された。

### 作ったファイル

**`backend/app/services/article_generator.py`（新規）**

```python
def generate_summary(host_text, guest_text):
    genai.configure(api_key=os.environ["GEMINI_API_KEY"].strip())
    model = genai.GenerativeModel("gemini-2.5-flash")  # ← 最終的にこれで動いた

    prompt = f"""
    ホスト（鶴岡）の発言：{host_text}
    ゲスト（大悟）の発言：{guest_text}
    以下のJSON形式のみで返してください...
    """
    response = model.generate_content(prompt)
    return json.loads(response.text.strip())
```

**`backend/requirements.txt`** に `google-generativeai` を追加。

## 4. Secret Manager導入

### なぜやったか

最初はCloud RunのEnvironment Variables（環境変数）にAPIキーを平文で保存していた。

**問題点：**
```bash
gcloud run services describe vibe-cast-backend
```
↑このコマンドを打つと、APIキーが **丸見え** になってしまう。

GCPのSecret Manager（秘密情報の金庫）に移行した。

### 仕組み

```
Secret Manager（金庫）
  ├── OPENAI_API_KEY = "sk-proj-..."
  └── GEMINI_API_KEY = "AIza..."
         ↓
Cloud Run（起動時にSecretを読み込む）
         ↓
Pythonコード（os.environ["OPENAI_API_KEY"]で取得）
```

アプリのコードからは `os.environ["OPENAI_API_KEY"]` と書けばいいだけで、
実際のキーの値はSecret Managerが管理してくれる。

### 正しい登録コマンド（Windowsでの注意あり）

**Windowsのechoは使ってはいけない。** `echo "sk-..."` とするとキーの末尾に
`\r\n`（改行コード）が混入してしまい、APIがエラーを返す。

**正しい方法（PowerShell Byte方式）：**
```powershell
$key = "sk-proj-..."
[System.Text.Encoding]::UTF8.GetBytes($key) | Set-Content -Path temp_key.bin -Encoding Byte
gcloud secrets versions add OPENAI_API_KEY --data-file=temp_key.bin --project vibe-cast-tsurudai
Remove-Item temp_key.bin
```

---

## 今日のトラブルと教訓

### トラブル1：APIキーに`\r\n`が混入してAPIキーがログに露出

**原因：**
`echo "sk-..."` でSecretに登録すると、Windowsの改行コード（`\r\n`）が末尾に付く。

**症状：**
```
Illegal header value b'Bearer sk-proj-...\r\n'
```
HTTPヘッダーに改行は入れられないのでエラーになる。
さらにスタックトレース（エラーの詳細ログ）にAPIキーが **平文で露出** してしまった。

**対策：**
1. 上記のPowerShell Byte方式で再登録
2. コード側でも `.strip()` を追加して防御

```python
api_key=os.environ["OPENAI_API_KEY"].strip()  # 念のため前後の空白・改行を除去
```

---

### トラブル2：Cloud Runの環境変数の型競合

**原因：**
最初に `--set-env-vars OPENAI_API_KEY=sk-...` で環境変数として設定していた。
あとで `--set-secrets OPENAI_API_KEY=OPENAI_API_KEY:latest` でSecretに変えようとしたら：

```
Cannot update environment variable to the given type
```

**理由：**
Cloud Runは「普通の環境変数」と「Secretからの環境変数」を別の型として管理している。
すでに普通の環境変数として登録されているキー名をSecretに変えることができない。

**解決策：**
```bash
# 一度全部クリアしてから
gcloud run deploy vibe-cast-backend \
  --set-secrets OPENAI_API_KEY=OPENAI_API_KEY:latest,GEMINI_API_KEY=GEMINI_API_KEY:latest \
  --clear-env-vars \
  ...
```

---

### トラブル3：Geminiのモデル名が次々と使えない

これが今日一番しんどかった。

| 試したモデル名 | 結果 |
|---|---|
| `gemini-1.5-flash` | v1beta（旧バージョン）非対応でエラー |
| `gemini-2.0-flash-lite` | 新規ユーザー向けに廃止済み |
| `gemini-2.0-flash` | 新規ユーザー向けに廃止済み（最新ログのエラー） |
| `gemini-2.5-flash` | ✅ **動いた！** |

エラーメッセージ：
```
google.api_core.exceptions.NotFound: 404
This model models/gemini-2.0-flash is no longer available to new users.
Please update your code to use a newer model.
```

Googleが古いモデルを順次廃止しており、新規ユーザーは使えなくなっていた。
`gemini-2.5-flash` に変えたら一発で動いた。

## 5. セッションID同期（episode_id）

### 問題

元々 `EPISODE_ID` はページを開いた瞬間の日付で生成していた：

```typescript
const EPISODE_ID = new Date().toISOString().slice(0, 10).replace(/-/g, '');
// → "20260403"
```

ホストとゲストが**独立してIDを生成**するため、日付が変わったりリロードしたりするとIDが噛み合わなくなる。

**具体的な問題：**
- ホストが23:59にページを開く → `20260403`
- ゲストが0:01にページを開く → `20260404`
- 文字起こしが噛み合わない

### 解決策

**ホストが録音開始時にセッションIDを生成して、WebSocket（シグナリングサーバー）経由でゲストに配信する。**

```
ホストが「録音開始」を押す
  ↓
"20260403183042" のようなIDを生成（日付+時刻）
  ↓
WebSocketで {"type": "session_id", "id": "20260403183042"} を送信
  ↓
シグナリングサーバーがゲストに転送（バックエンド変更不要）
  ↓
ゲストのepisodeIdが自動でホストと同じ値に上書きされる
```

**シグナリングサーバーを変更しなくてよかった理由：**
すでに「受け取ったメッセージを同じ部屋の全員に転送する」仕様になっていたから。
`session_id` という新しい種類のメッセージを追加するだけでよかった。

### 変更したファイル

**`frontend/src/hooks/useWebRTC.ts`**
- `onSessionId` コールバックを追加
- `session_id` メッセージを受信したらコールバックを呼ぶ
- `sendSessionId(id)` 関数を追加（ホストが録音開始時に呼ぶ）

**`frontend/src/pages/RecordingPage.tsx`**
- `EPISODE_ID` を定数から `useState('')` に変更
- ホストの録音開始時に `generateEpisodeId()` → `sendSessionId()` を呼ぶ
- ゲストは `setEpisodeId` を `onSessionId` コールバックとして渡す

## 6. 録音をホスト限定に

### 問題

- ゲストもラジオボタンを「ホスト」に切り替えれば録音ボタンが押せてしまう
- 接続後でもロール変更が自由にできた

### 解決策

**接続ボタンを押した瞬間（`localStream` が設定された瞬間）にラジオボタンを `disabled` にする。**

```typescript
<input
  type="radio"
  value="host"
  checked={role === 'host'}
  onChange={() => setRole('host')}
  disabled={!!localStream}  // 接続後は変更不可
/>
```

これで「接続する前にロールを選ぶ → 以降は変更不可」という流れが強制される。
ゲストには録音ボタンの代わりに「🎙 録音はホストが操作します」と表示される。

## 最終的なシステム構成

```
ブラウザ（Firebase Hosting: vibe-cast-tsurudai.web.app）
  │
  ├─── WebRTC P2P通話（ホスト↔ゲスト）
  │
  ├─── 録音 → GCSアップロード
  │         gs://vibe-cast-tsurudai/episodes/{日付}/raw/{role}_track.webm
  │
  ├─── [文字起こしボタン] → Cloud Run /ai/transcribe
  │         → GCSから音声DL → Whisper API → テキスト返却
  │
  └─── [要約ボタン] → Cloud Run /ai/summarize
            → Gemini API → JSON返却 → 画面に表示

Cloud Run（FastAPI: vibe-cast-backend）
  ├─ /audio/upload  → GCS保存
  ├─ /ai/transcribe → Whisper API（文字起こし）
  └─ /ai/summarize  → Gemini API（要約・構造化）

Secret Manager（APIキーを安全に管理）
  ├─ OPENAI_API_KEY
  └─ GEMINI_API_KEY

GCS（音声ファイル保存）
```
