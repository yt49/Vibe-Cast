## このアプリって何？

二人でリモートでポッドキャストを収録するWebアプリ。
ZoomとかDiscordを使わず、ブラウザだけで通話・録音・GCS保存まで完結する。

**URL:** https://vibe-cast-tsurudai.web.app

---

## 全体の構成（超シンプル版）

```
あなたのブラウザ ─── シグナリングサーバー ─── 大悟のブラウザ
     │                   （Cloud Run）               │
     └──────── WebRTC（直接P2P通話） ────────────────┘
     │                                               │
  自分の音声を録音                              自分の音声を録音
     │                                               │
     └──────────── GCS（音声ファイル保存） ───────────┘
```

**3つのサービスが動いている：**

| サービス名 | 役割 | 場所 |
|-----------|------|------|
| Firebase Hosting | フロントエンド（画面）を配信 | Google Firebase |
| Cloud Run | バックエンド（シグナリングサーバー） | Google Cloud |
| Cloud Storage（GCS） | 録音ファイルの保存場所 | Google Cloud |

## 専門用語の解説

### WebRTC（ウェブアールティーシー）
ブラウザ同士が**直接**音声・映像をやりとりする仕組み。
Zoomとかも内部ではこれを使ってる。
「P2P（ピアツーピア）」とも言う。サーバーを経由せず直接つながるのが特徴。

### シグナリングサーバー
WebRTCで直接つながる前に「俺ここにいるよ」「了解、こっちはここ」と
**待ち合わせ情報を交換するための中継点**。
通話が始まったらシグナリングサーバーの仕事は終わり。
→ 今回はCloud Runで動いてる。

### STUN/TURNサーバー
- **STUN**：「あなたのIPアドレスはこれですよ」と教えてくれるサーバー。無料。
- **TURN**：会社ネットワーク等でSTUNが使えない時に**中継**してくれるサーバー。
  今回はSTUNだけ設定済み（同一Wi-FiやシンプルなWi-Fiなら繋がる）。
  異なるネットワーク間で繋がらなかったらTURNを追加する（issue #6）。

### Cloud Run（クラウドラン）
Googleのサービス。**プログラムをサーバーなしで動かせる**。
「コンテナ」と呼ばれるパッケージにプログラムを入れて、Googleのサーバーで動かす。
アクセスがないときはコストゼロ、アクセスがきたら自動起動する。

### GCS（Google Cloud Storage）
GoogleのDropboxみたいなやつ。ファイルをクラウドに保存できる。
今回は録音した音声ファイル（.webm）を保存している。
バケット名：`vibe-cast-tsurudai`
保存先：`gs://vibe-cast-tsurudai/episodes/20260402/raw/host_track.webm`

### Firebase Hosting
Googleの静的サイトホスティングサービス。
ReactでビルドしたHTMLをここに置くと、世界中からURLでアクセスできる。
無料枠で十分動く。

### WebSocket（ウェブソケット）
通常のHTTP通信は「リクエスト→レスポンス」の一方通行だが、
WebSocketは**双方向リアルタイム通信**ができる。
シグナリングサーバーとの通信に使っている。
`ws://`が通常版、`wss://`が暗号化版（Cloud Runはwssが必要）。

### MediaRecorder
ブラウザ内蔵の録音機能。マイクの音声をファイルに変換できる。
今回は`.webm`形式で録音している。

### 署名付きURL（Signed URL）
GCSに直接アップロードするための一時的なURL（15分だけ有効）。
ブラウザから直接GCSにアクセスするには認証が必要だが、
バックエンドが「この人は信頼できる」と証明した一時URLを発行することで安全にアップロードできる。
→ 今回はバックエンドがプロキシとして直接受け取ってGCSに上げる方式にした。

### 1. GitHub MCP 接続

**MCPとは何か：**
Claude（AI）が外部サービス（GitHub等）を操作できるようにする仕組み。
Claude Code のターミナルで設定する。

**やったこと：**
```
claude mcp add --transport http github https://api.githubcopilot.com/mcp/ \
  --header "Authorization: Bearer <GitHubトークン>"
```
GitHubのPersonal Access Tokenを取得してClaudeに接続した。

### 5. バックエンド（Cloud Run）デプロイ

バックエンドは Python + FastAPI で書かれている。

**Dockerfileとは：**
「このプログラムを動かすための環境を作る手順書」。
Cloud RunはこのDockerfileを読んでプログラムを動かす。

```dockerfile
FROM python:3.11-slim    # Python 3.11の環境を使う
WORKDIR /app             # 作業フォルダを /app に設定
COPY requirements.txt .  # 必要なライブラリのリストをコピー
RUN pip install ...      # ライブラリをインストール
COPY . .                 # プログラムをコピー
CMD ["uvicorn", ...]     # サーバーを起動
```

**デプロイコマンド：**
```
cd backend
gcloud run deploy vibe-cast-backend \
  --source . \
  --region asia-northeast1 \
  --allow-unauthenticated
```

`--allow-unauthenticated` = 誰でもアクセスできる（認証不要）

**デプロイ後のURL：**
`https://vibe-cast-backend-905541599300.asia-northeast1.run.app`

**詰まったポイント：**
- フロントの WebSocket URL が `ws://localhost:8000` のままだった
  → `wss://vibe-cast-backend-905541599300...` に変更が必要だった
- Cloud Runは `https` しか受け付けないので `wss://`（暗号化WebSocket）が必要

### 6. フロントエンド（Firebase Hosting）デプロイ

**やったこと：**
1. `npm install -g firebase-tools` でFirebase CLIをインストール
2. `firebase login` でGoogleアカウントと連携
3. Firebase コンソール（console.firebase.google.com）で

- 新規プロジェクトを作ろうとしたら403エラー（権限なし）

### 7. GCSアップロード機能の実装

**流れ：**
```
録音停止ボタンを押す
  ↓
ブラウザが音声データ（Blob）を取得
  ↓
バックエンド（Cloud Run）の /audio/upload に POST送信
  ↓
バックエンドが GCS に保存
  ↓
「✅ アップロード完了！」が表示される
```

**バックエンドに追加したコード（audio.py）：**
```python
@router.post("/audio/upload")
async def upload_audio(file, role, episode_id):
    # GCSクライアントを作成
    client = storage.Client()
    bucket = client.bucket("vibe-cast-tsurudai")
    blob = bucket.blob(f"episodes/{episode_id}/raw/{role}_track.webm")
    # ファイルをアップロード
    blob.upload_from_string(content, content_type="audio/webm")
```

**詰まったポイント：**
- バケット名を `vibe-cast-tsurudai-audio` と書いてたが実際は `vibe-cast-tsurudai` だった
  → アップロード「完了」と表示されてたが実際は失敗してた（エラー処理がなかった）
  → バケット名修正・再デプロイで解決

### 8. 切断ボタンの修正

**なぜ動かなかったか：**
ボタンの表示切り替えは `localStream`（マイクの音声ストリーム）の有無で判定していた。
`disconnect()` を呼んでも `localStream` をリセットしていなかったので、
ボタンが「切断する」のまま変わらなかった。

**修正内容：**
```typescript
const handleDisconnect = () => {
  localStream?.getTracks().forEach((t) => t.stop()); // マイクを止める
  setLocalStream(null);  // localStreamをリセット
  disconnect();          // WebRTC・WebSocket を切断
};
```
